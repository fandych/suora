# Suora 当前已知缺口与待补强项

> 更新日期：2026-04-30  
> 口径：只记录当前仓库里能较稳妥确认的缺口，不再沿用旧 PRD 时代的大而全勾选表。

## 1. 使用说明

这不是一份“所有未来想法”的收集箱，而是面向维护者的保守清单。只有满足以下任一条件的事项，才应该写进来：

- 当前代码里没有对应实现痕迹。
- 当前专项审计文档已明确标注为 future pass。
- 当前测试/构建配置能直接证明这一能力仍未闭环。

## 2. 高优先级缺口

### 2.1 缺少 Electron 原生端到端测试闭环

当前 Playwright 配置通过 Vite dev server 启动 renderer smoke 测试，这对前端回归有价值，但它还不是 Electron 窗口级自动化。

影响：

- 主进程与 preload bridge 的真实桌面行为仍主要依赖单测与手工验证。
- 打包产物、窗口生命周期、桌面权限边界缺少专门的 e2e 回归保护。

建议：

- 增补直接启动 Electron 的自动化测试链路。
- 把关键 IPC、安全提示和窗口行为纳入桌面级 smoke 场景。

### 2.2 Pipeline 仍缺少更高级的编排能力

当前 pipeline 已支持多步执行、重试、超时、`runIf`、变量导出和历史，但从现有专项审计与代码面看，以下能力仍未形成稳定实现：

- 并行 fan-out / fan-in 步骤
- typed step output
- 变量 schema 校验
- 从失败步骤继续 rerun
- 不同执行记录的 diff 对比

影响：

- 复杂自动化仍偏串行。
- 较长 pipeline 在复跑和调试时成本更高。

### 2.3 Chat 运行时的恢复与诊断还可继续增强

当前聊天链路已经有取消、重试、sanitization、tool event 和 pipeline chat 命令，但仍有一批缺口在专项清单中被标记为后续项：

- stream heartbeat / health event
- 更明确的 empty-response recovery UI
- 更强的 pipeline 命令模糊匹配与歧义引导
- 更完整的本地化命令帮助

影响：

- 用户在 provider 卡顿、空回复或命令轻微拼写错误时的恢复成本仍偏高。

## 3. 中优先级缺口

### 3.1 Agent 治理能力还不算闭环

当前 agent 已支持版本记录、分析、技能和工具配置，但仍缺少更强的治理面：

- 版本差异对比视图
- 更严格的导入校验与风险提示
- 面向 agent 的批量评测 harness

影响：

- agent 配置演进虽然可追踪，但还不够容易比较和评估。

### 3.2 文档与实现之间还没有自动漂移检测

这轮文档刷新说明了一个现实问题：仓库中已有不少文档会随着功能扩展而失真，但当前没有自动校验机制帮助发现漂移。

影响：

- 路由、provider、settings 分区、内置 agent 名单等事实容易在文档中老化。

建议：

- 逐步建立面向文档锚点的检查脚本或人工更新清单。
- 尽量避免在文档里固定容易漂移的总数类信息。

## 4. 产品形态层面的非现成能力

以下能力不应被误写成“当前已经有”：

### 4.1 IDE 内联补全与编辑器原生插件形态

当前仓库不是 VS Code、JetBrains 或 Neovim 的原生扩展项目，也没有实现内联代码补全主路径。

### 4.2 Web SaaS 或移动端客户端

当前仓库的主交付形态是 Electron 桌面应用，不是 Web 多人协作服务，也不是移动端原生客户端。

### 4.3 全量桌面发布自动化闭环

仓库已有构建与打包脚本，但不应把它描述成“发布链路已经完全闭环并长期验证完毕”，除非后续把签名、更新、发布验收与桌面 smoke 测试整合到稳定流程中。

## 5. 当前不建议继续沿用的旧写法

请不要再维护类似下面这种高风险写法：

- 给每个模块写一个巨大的“已实现百分比”表格
- 把旧规划能力全部逐项标为已实现或未实现
- 在未经复核的情况下写死工具总数、IPC 总数、平台总数、测试总数
- 把未来优化项写成已经完成的功能

## 6. 推荐的维护方式

后续维护这份文件时，优先基于以下锚点判断缺口：

- `playwright.config.ts`
- `vitest.config.ts`
- `src/App.tsx`
- `src/store/appStore.ts`
- `src/services/aiService.ts`
- `docs/AGENT_PIPELINE_CHAT_OPTIMIZATION_100.md`
- `docs/AGENT_PIPELINE_CHAT_EXECUTION_CAPABILITY_100_ZH.md`

如果某项能力只有规划文档里提到、代码里没有落点，就应继续放在“待补强”而不是“已实现”。
