# Suora 产品范围与需求基线

> 版本：implementation-aligned baseline  
> 更新日期：2026-04-30  
> 状态：维护中

本文档不再作为早期概念 PRD 使用，而是作为“当前代码对应的产品范围与需求基线”。如果实现发生变化，应优先以代码为准，再同步更新本文档。

## 1. 产品定位

Suora 是一款基于 Electron 的本地优先 AI 工作台。当前代码体现出的核心定位不是单一聊天窗口，也不是 IDE 内联补全插件，而是一个把以下能力放进同一桌面壳中的工作环境：

- 多模型与多提供商接入
- 多 agent 协作与定制
- skill 包管理与能力装配
- 文档工作区与聊天上下文联动
- agent pipeline 编排与执行历史
- 定时任务与自动触发
- 外部消息渠道接入
- MCP 服务配置与集成
- 本地数据管理、安全边界与运行日志

## 2. 产品原则

### 2.1 Local-first

用户的会话、文档树、agent、skill、pipeline、渠道配置和大部分运行元数据都应默认保存在本地，而不是要求云端后端才能使用。

### 2.2 BYOK 与多模型自由切换

用户应能够自行配置 API Key、Base URL 和默认模型，并在同一工作台中切换不同 provider。

### 2.3 Agent-first 执行体验

聊天、pipeline、timer 和渠道回复都应围绕 agent 配置运行，而不是把模型调用写死在单一路径上。

### 2.4 安全优先

所有特权能力必须通过 Electron preload bridge 暴露；安全存储失败时必须向用户显式提示；文件系统、shell 与工具执行应支持限制与确认。

## 3. 目标用户与主要场景

### 3.1 目标用户

- 需要在本地桌面环境统一管理多个模型与 agent 的高级用户
- 需要把 AI 与文档、任务流、外部消息渠道联动起来的知识工作者
- 需要用可配置工具、skills、MCP 扩展桌面 AI 能力的开发者与技术团队

### 3.2 主要场景

- 在聊天工作区中完成带工具调用的复杂问答
- 选择不同 agent 处理代码、写作、研究、安全、数据分析或运维任务
- 把文档内容组织为可检索、可引用、可回链的知识上下文
- 把多步任务保存为 pipeline，并从聊天、界面或定时任务中触发执行
- 将外部消息渠道接入工作台并绑定专用回复 agent
- 通过 MCP 把外部服务能力引入 agent 运行时

## 4. 当前产品范围

当前顶层模块由 `src/App.tsx` 中的 Hash Router 定义。

| 模块 | 路由 | 当前需求基线 |
| --- | --- | --- |
| Chat | `/chat` | 支持会话管理、流式回复、附件、模型与 agent 选择、工具事件显示、失败重试与 pipeline 命令 |
| Documents | `/documents` | 支持文档组、嵌套目录、Markdown、图谱、回链、Mermaid、数学公式与聊天上下文选择 |
| Pipeline | `/pipeline` | 支持多步 agent 工作流、重试、超时、`runIf`、变量导出、可视化预览、执行历史与导入导出 |
| Models | `/models/:view` | 支持 provider 配置、模型管理、比较视图与默认模型管理 |
| Agents | `/agents` | 支持内置与自定义 agent、prompt、skills、工具限制、记忆、版本与统计 |
| Skills | `/skills/:view` | 支持已安装 skills、注册表浏览、来源管理、`SKILL.md` 编辑、资源树、导入与导出 |
| Timer | `/timer` | 支持 Once、Interval、Cron 三类任务，以及通知、agent prompt、pipeline 执行三类动作 |
| Channels | `/channels` | 支持渠道配置、reply agent、auto reply、消息历史、用户列表、健康与调试面板 |
| MCP | `/mcp` | 支持 MCP 服务器配置、连接状态与能力接入 |
| Settings | `/settings/:section` | 支持 general、security、voice、shortcuts、data、logs、system 七个设置分区 |

## 5. 关键功能需求

### 5.1 Chat

- 必须支持多会话与持久化会话状态。
- 必须支持流式输出与工具调用事件展示。
- 必须允许用户切换模型、切换 agent，并保留当前上下文。
- 必须支持 Markdown 渲染，包含代码块、表格、列表、数学公式与 Mermaid。
- 必须支持文件附件，并在失败重试时保留附件上下文。
- 必须支持 `/pipeline` 相关命令，至少覆盖列出、运行、状态、历史与取消。

### 5.2 Documents

- 必须支持文档组、目录和文档节点的树状管理。
- 必须支持 Markdown 编辑与渲染。
- 必须支持回链、引用、搜索与图谱浏览。
- 必须支持把文档作为聊天上下文提供给当前会话或当前任务。

### 5.3 Models 与 Providers

- 必须支持多个 provider 共存。
- 必须支持自定义 API Key 与 Base URL。
- 必须对模型配置进行校验，并在连接失败时反馈可理解的错误。
- 必须在统一模型层之上支持文本生成与流式响应。

### 5.4 Agents

- 必须内置至少一个通用 assistant 与多个专业 agent。
- 必须允许用户创建、复制、导入、导出和删除自定义 agent。
- Agent 配置必须覆盖系统提示词、模型偏好、skills、工具白名单/黑名单、权限模式、温度与 token 上限等要素。
- Agent 运行时必须能结合 skills、memories 与 permission 配置生成系统上下文。

### 5.5 Skills

- Skills 必须以 `SKILL.md` 为核心描述文件。
- 必须支持本地导入、目录导入、导出和资源树管理。
- 必须支持已安装视图、来源视图和注册表/市场浏览视图。
- Skill 编辑器必须提供预览或辅助面板，而不是仅存储原始文本。

### 5.6 Pipeline

- 必须支持多步骤执行，并允许每一步绑定 agent。
- 必须支持重试、步级超时、输出变换、变量导出和 `runIf` 条件执行。
- 必须保存执行记录、步骤状态和历史元数据。
- 必须支持导入、导出、Mermaid 预览与从聊天触发运行。

### 5.7 Timer

- 必须支持一次性、间隔和 Cron 三种调度形式。
- 必须支持桌面通知、agent prompt 执行和 pipeline 执行三类动作。
- 必须向用户展示下一次执行时间、最近运行状态和历史记录。

### 5.8 Channels

- 必须支持当前类型系统中定义的渠道平台矩阵，而不是只限单个平台。
- 每个渠道必须允许绑定 reply agent、是否自动回复以及平台相关配置。
- 渠道模块必须提供健康、消息、用户和调试四类观察面。
- 渠道相关能力必须经由主进程与 preload bridge 暴露，不能在 renderer 直接访问敏感系统接口。

### 5.9 MCP

- 必须支持 MCP 服务器的配置与状态管理。
- 必须让 agent 运行时可以识别并调用已连接 MCP 能力。
- 必须在设置与集成界面中向用户展示连接状态。

### 5.10 Settings、Data 与 Security

- 必须支持语言、主题、字体和强调色等界面偏好。
- 必须支持数据导入、导出、备份与清理操作。
- 必须支持日志、崩溃信息与运行诊断查看。
- 必须在 Secure Storage 不可用时显式警告用户密钥只会驻留内存。
- 必须支持工具确认、目录允许列表、危险 shell 模式拦截等安全策略。

## 6. 非功能需求

### 6.1 架构要求

- Electron 必须启用 `contextIsolation`。
- Renderer 不得直接暴露 Node.js 特权能力。
- 顶层模块应按需 lazy load，避免一次性加载所有工作区视图。

### 6.2 可维护性

- 核心状态应集中在单一持久化 store 中管理。
- 类型定义应统一维护在共享类型层，而不是散落在组件内部。
- 重要运行能力必须具备单元测试或组件测试覆盖。

### 6.3 文档要求

- 文档中的模块、路由、provider、settings 分区和内置 agent 名称必须以当前代码为准。
- 不应长期硬编码容易漂移的数量，例如 IPC 通道总数、工具总数或覆盖率百分比，除非更新前已重新验证。

## 7. 当前非目标

以下内容不属于当前仓库已经交付的主目标，不应在文档中描述为现成能力：

- 作为 IDE 内联补全插件运行
- 作为 Web SaaS 提供多人协作后端
- 提供移动端原生客户端
- 对所有 provider 暴露完全一致的能力矩阵
- 用单一文档替代代码中的真实实现细节
