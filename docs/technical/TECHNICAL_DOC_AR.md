# الوثائق التقنية الخاصة بـ Suora

هذا المستند هو مرجع تقني مبني على التنفيذ الحالي في المستودع. وهو موجه للمساهمين والمسؤولين عن الصيانة، ولا يصف إلا البنى الموجودة فعليًا في الشفرة.

## 1. نظرة عامة على النظام

Suora عبارة عن منصة عمل محلية للذكاء الاصطناعي مبنية على Electron. الوحدات الرئيسية المتوفرة حاليًا هي:

- Chat
- Documents
- Pipeline
- Models
- Agents
- Skills
- Timer
- Channels
- MCP
- Settings

يتبع التطبيق نهج local-first. يتم تخزين حالة المستخدم والجلسات وأشجار المستندات وإعدادات الوكلاء وإعدادات النماذج ومعظم بيانات التشغيل الوصفية محليًا عبر طبقة حفظ تعتمد على IPC.

## 2. بنية التشغيل

ينقسم وقت التشغيل إلى ثلاث طبقات.

| الطبقة | المسؤولية |
| --- | --- |
| Electron Main Process | يدير نظام الملفات ومساعدات fetch الشبكية وSecure Storage وshell وruntime القنوات وhandlers الخاصة بـ IPC |
| Preload Bridge | يعرّض واجهة `window.electron` مبنية على allowlist مع context isolation |
| React Renderer | يرسم واجهة العمل، ويدير الحالة عبر Zustand، ويقوم بتنظيم الذكاء الاصطناعي والمستندات وخطوط الأنابيب والقنوات والإعدادات |

يستخدم الـ renderer مسار Hash Router ويحمّل الوحدات بشكل lazy.

## 3. بنية المستودع

المستودع الحالي منظم حول غلاف Electron وتطبيق React منظم حسب الميزة.

```text
electron/
  main.ts          العملية الرئيسية لـ Electron و handlers الخاصة بـ IPC
  preload.ts       جسر preload معزول
  channelService.ts
  database.ts

src/
  App.tsx          تهيئة الموجّه والتهيئة العامة
  main.tsx         نقطة دخول renderer
  index.css        رموز الثيم العامة وأنماط الواجهة
  components/      الوحدات الوظيفية والواجهة المشتركة
  hooks/           React hooks
  services/        AI وstorage وi18n وpipelines وchannels وdocuments
  store/           Zustand store وslices
  types/           الأنواع المشتركة

docs/
  user/            وثائق المستخدم
  technical/       المراجع التقنية

e2e/
  اختبارات Playwright الطرفية
```

## 10. نموذج IPC والأمان

يحافظ Suora على context isolation في Electron ويوجه العمليات المميزة عبر preload bridge.

### أهم خصائص الأمان الحالية

- لا يصل renderer مباشرة إلى APIs الخاصة بـ Node.js
- لا يعرض preload إلا سطح invoke/on/send مبنيًا على allowlist
- تظهر أعطال secure storage كتحذيرات في الواجهة
- يمكن عزل الوصول إلى نظام الملفات بنمط sandbox
- يمكن للمستخدم تحديد الأدلة المسموح بها
- يمكن حظر أنماط shell الخطيرة
- يمكن طلب تأكيد قبل تنفيذ الأدوات

### سلوك Secure Storage الحالي

يحاول التطبيق أولًا حفظ مفاتيح API في التخزين الآمن الخاص بنظام التشغيل. إذا لم يكن هذا التخزين متاحًا أو فشل التشفير، تحذر الواجهة من أن المفاتيح ستبقى في الذاكرة فقط ويجب إدخالها مجددًا بعد إعادة التشغيل.

## 12. ملاحظات الصيانة

عند تحديث الوثائق التقنية لهذا المستودع، أعط الأولوية دائمًا للحقائق المؤكدة من الشفرة بدلًا من الأوصاف الموروثة. أكثر نقاط الارتكاز موثوقية هي:

- المسارات الفعلية في `src/App.tsx`
- الوكلاء المدمجون الفعليون في `src/store/appStore.ts`
- أنواع المزوّدين الفعلية في `src/services/aiService.ts`
- أقسام الإعدادات الفعلية في `src/components/settings/SettingsLayout.tsx`

ما لم تكن قد راجعت الشفرة قبل لحظات، تجنب تثبيت أرقام معرّضة للانجراف في الوثائق مثل العدد الإجمالي لقنوات IPC أو العدد الإجمالي للأدوات.
